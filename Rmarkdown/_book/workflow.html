6 Workflow

6.1 Git Repository Set-up

Each repository should start with these branches

  • master

  • This is the master code branch
    • All code in this repository has been tested and is solid
    • This code has been ‘released’

  • develop

    • This is the develop code branch
    • The develop branch is created from the master branch
    • This code is has also been tested and is solid but is pre-release

  • features/

    • This is a folder for feature code branches
    • Feature branches are created from the develop branch
    • This code is in development
    • It may be pre-tested right now and is still in progress

6.2 Workflow

6.2.1 Programmer I

  1. Create a feature branch from the develop branch
    1. E.g. git checkout -b features/NameOfFeature_JIRA#
    2. Notice that that a JIRA ticket number is attached at the end so it’s easy to link back to project management and see the initial development request
  2. Make changes and edits to code
    1. Make commits whenever there’s a major change to the code
    2. Remember commits shouldn’t have ‘and’ in them

    3. Remember commits should explain the why not the what

      git add name_of_file
git commit -m "commit explaining why not the what"
  3. Write unit tests
  4. Once code is written and tests are passing submit at Pull Request to the develop branch

6.2.2 Programmer II

  1. Review the Pull Request and suggest changes/edits or complete the request
    1. Code gets merged into develop

  2. Pull the develop branch

    git checkout develop
git pull

  3. Merge the develop branch into any feature branches currently being worked on

6.2.3 Finalizing code

  1. Once the iterative process between developers and the code is complete and tested it can be merged into the master branch

6.2.5 In-depth development workflow

  1. A develop branch is created from master
  2. Feature branches are created from develop
  3. When a feature is complete a pull request is made for a potential merge into the develop branch
  4. This request goes through a code review and can be approved/rejected by the technical lead
  5. When develop branch is complete for a given cycle, a release branch is created from develop
  6. When the release branch is tested and deemed ready for production it is merged into develop and master
  7. If an issue in master is detected a hotfix branch is created from master
  8. Once the hotfix is complete a pull request is made for a potential merge to both the develop and master.
